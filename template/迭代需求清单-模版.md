# 迭代需求清单（模版）

> **用途**：作为迭代需求输入源，用于驱动 `docs/prd/V{版本号}/` 下 PRD 与模块需求文档生成。  
> **建议命名**：`ATO_V{X.Y.Z}-需求清单.md`  
> **建议存放**：`docs/prd/V{X.Y.Z}/`

---

## 1. 基本信息

- **版本号**：V{X.Y.Z}
- **迭代名称**：{示例：V1.0.2 需求迭代}
- **维护人**：{产品姓名}
- **创建日期**：YYYY-MM-DD
- **状态**：草稿 / 评审中 / 已确认

---

## 2. 字段说明

| 字段 | 是否必填 | 说明 |
|------|----------|------|
| 业务域 | 是 | 如：系统设置、自动化开发、版本用例开发、自动化应用、测试工具 |
| 模块 | 是 | 功能模块名称，建议与 `.cursor/rules/requirements-module-split.mdc` 对齐 |
| 需求类型 | 是 | 新增 / 优化（用于区分新能力建设与存量能力改进） |
| 一级需求描述 | 是 | 模块级主要需求点（用户可见能力） |
| 二级需求描述 | 是 | 对一级需求的细化，可写子功能/子流程 |
| 需求动机 | 是 | 为什么要做（问题、价值、目标） |
| 是否涉及UI | 是 | 是 / 否 |
| 优先级（高中低） | 是 | 高 / 中 / 低 |
| 版本号 | 是 | 与文档版本保持一致 |
| 是否已实现 | 是 | 是 / 否 / 部分实现 |
| 需求ID（推荐） | 否 | 建议格式：`REQ-V{X.Y.Z}-001`，便于追踪 |
| 关联页面/入口（推荐） | 否 | 如：自动化开发-项目管理；或菜单路径 |

## 4. 使用约束（建议）

1. 模块字段应优先对齐 `.cursor/rules/requirements-module-split.mdc`。  
2. 生成需求文档时，默认输出到 `docs/prd/V{X.Y.Z}/`。  
3. 未收到“允许合入全量（docs/requirements）”明确指令前，不得写入 `docs/requirements/`。  
4. 若“是否涉及UI=否”，仍需补充业务流程、规则与影响范围。  
5. 每次迭代结束后，保留此清单用于需求追溯与验收对账。
